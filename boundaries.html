Scope

ZLAR only governs actions that pass through it.

ZLAR is a doorway, not a magic claim about every AI action everywhere. This page says what the doorway covers and what it does not.

What ZLAR governs

Actions routed through the doorway.

An action is governed when it flows through a ZLAR interception surface before it takes effect. On that path, ZLAR checks the signed rule and records the decision.

Core boundary

ZLAR governs routed/intercepted action surfaces only.

Current proof path

The Codex claim is narrow on purpose.

Release state: ZLAR v3.3.76 on GitHub. The current release surfaces fixture-contained active-profile selection evidence in the proof pack and proof-smoke report for the first `records.write` terminal: the sample contract records the explicit pinned runtime profile selected inside the local disposable proof harness while preserving no persistent runtime-profile install, no hook activation, no live profile check, and no `--latest` selection. It preserves the bounded `records-write-terminal-proof` proof path for recognized `records.write` acceptance plus missing or unrecognized receipt refusal before mutation, plus the public claim guard: approval routing is described as configured-channel behavior, not unconditional phone or Telegram availability. Release-by-release detail lives in GitHub Releases and the core changelog, not as copied authority on this page. Boundary: no live approval-channel delivery or health, no production authority, no current-machine governance, no external attestation, no sovereign recognition, no all-MCP governance, and no unrouted-surface coverage. A private-by-default non-Vincent verifier request has been sent; no public external attestation is claimed in this repo. The verifier environment JSON report `zlar-verifier-env-report-v0.json` creates no public external attestation. The current Codex adapter is Telegram-silent by default with `ZLAR_TELEGRAM_DISABLED=1`; that does not prove Telegram health. The public-copy guard enforces these non-claims in CI; that guard is CI evidence only and adds no runtime authority.

Unrouted client surfaces and model text.

Non-claims

What the sample does not prove.

Explicit non-claims

  • Unrouted shell/filesystem/browser/app/network/model-reasoning/final-text surfaces are not claimed as governed by this proof path.
  • Direct MCP registrations that bypass the ZLAR route are outside this evidence path.
  • /contest is not implemented.
  • A private-by-default non-Vincent verifier request has been sent; no public external attestation is claimed in this repo, and any private reply or later result remains bounded by verifier relationship, disclosure permission, and exact evidence returned.
  • A log records what happened. A ZLAR receipt records what counted as authorized effect. It does not prove the rule was correct or the AI's intention was good.
