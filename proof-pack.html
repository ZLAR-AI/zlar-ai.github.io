What the receipt says

A receipt settles the boundary.

The sample receipt points to a bounded evidence bundle. It records the action, the rule, the decision, the verification path, and the coverage report that says what ZLAR saw and what it did not see.

For developers and auditors, the technical artifact name is Proof Pack v0. For a first visitor, the important thing is simpler: a log records what happened; a receipt records what counted as authorized effect.

Developer verification

Verify the sample in Terminal.

These commands download the public sample artifacts, check the SHA-256 sidecar, and run the local verifier. 

mkdir zlar-proof-pack-v0-scratch
cd zlar-proof-pack-v0-scratch
mkdir -p evidence

curl -fsSLO https://zlar.ai/demo/proof-pack/README.md
curl -fsSLO https://zlar.ai/demo/proof-pack/proof-pack-manifest.json
curl -fsSLO https://zlar.ai/demo/proof-pack/SHA256SUMS
curl -fsSLO https://zlar.ai/demo/proof-pack/verify-proof-pack.mjs
curl -fsSLo evidence/governed-profile-coverage-report.json \
  https://zlar.ai/demo/proof-pack/evidence/governed-profile-coverage-report.json
curl -fsSLo evidence/governed-profile-coverage-report.txt \
  https://zlar.ai/demo/proof-pack/evidence/governed-profile-coverage-report.txt

shasum -a 256 -c SHA256SUMS
node verify-proof-pack.mjs

Current scope

Release state: ZLAR v3.3.78 on GitHub. The prepared, not-sent release-forward external verifier packet remains pinned to the `v3.3.76` proof-pack/proof-smoke release and records local dry-run evidence that future verifiers should preserve active-profile selection counts and the embedded `active_profile_selection` summary. The current release hardens verifier active-profile boundaries around that target: README and the coverage-map quickstart tell evaluators to preserve the `v3.3.76+` active-profile counts and embedded summary, the packet attestation template names those artifacts, and the verifier-kit runner says no external attestation is claimed unless a real non-operator runner fills, signs, or publishes the result. The underlying `v3.3.76` active-profile evidence still records no persistent runtime-profile install, no hook activation, no live profile check, and no `--latest` selection. It sends no new verifier request, contacts no verifier, creates no public external attestation, and does not prove production deployment, current-machine governance, live issuer status, key custody, revocation truth, sovereign recognition, enterprise readiness, or unrouted-surface coverage. The underlying proof path remains the bounded `records-write-terminal-proof` and proof-pack/proof-smoke sample contract for recognized `records.write` acceptance plus missing or unrecognized receipt refusal before mutation, and approval routing is described as configured-channel behavior, not unconditional phone or Telegram availability. Release-by-release detail lives in GitHub Releases and the core changelog, not as copied authority on this page. The downloadable sample bundle is bounded fake/scratch evidence for the safe routed-MCP proof path. It is not production deployment evidence and not external verifier attestation.

  • ZLAR governs routed/intercepted action surfaces only.
  • Safe Codex wording: "ZLAR can govern Codex CLI-invoked MCP tool calls when those MCP servers are routed through ZLAR."
  • Unrouted shell/filesystem/browser/app/network/model-reasoning/final-text surfaces are not claimed as governed by this proof path.
  • /contest is not implemented.
  • A private-by-default non-Vincent verifier request has been sent; no public external attestation is claimed in this repo, and any private reply or later result remains bounded by verifier relationship, disclosure permission, and exact evidence returned.
